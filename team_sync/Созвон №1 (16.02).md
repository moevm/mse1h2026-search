## Протокол встречи 
1. Дамп БД и индексирование документов
    - Дамп БД будет отправлен в среду — это MySQL.
    - Необходимость индексации документов — по желанию (предложено [Apache tika](https://tika.apache.org/)).
2. Режимы поиска, фильтры, языки
    - Фильтры для поиска: 
      - языки — русский, английский (далее смотреть, что будет в базе);
      - regexp;
      - сайт (область поиска*);
      - дата публикации (по умолчанию новые статьи более приоритетные).
3. Формат результата поиска
    - Формат результата — заголовок + дата + слова, использованные в запросе.
4. Ресурсы и нагрузка
    - Ограничения по ресурсам:
      - на сервере не будет GPU;
      - 8/16 Gb RAM;
      - 2-4 Cores;
      - с SSD проблемы — включить в анализ.
    - Количество одновременных пользователей:
      - 400-500;
      - в дальнейшем провести стресс-тестирование системы с целью выявления узких мест.
5. Авторизация и логи
    - Авторизация не является необходимостью. В перспективе возможно предусмотреть поддержку передачи JWT-токена в запросе с последующим извлечением user_id для реализации персонализации.
    - Предусмотреть логирование запроса, ответа и перехода пользователя с сохранением данных в PostgreSQL; админ-панель и отдельный интерфейс для просмотра логов не требуются.
6. Метрики качества
    - Основной метрикой является F1@k. Допускается добавление 1–2 дополнительных метрик (например, чувствительности и специфичности).
7. Макет и сценарии использования
    - Разделение на группы пользователей (абитуриент, студент, сотрудник) в целом нужно ТОЛЬКО для описания use caseов.

- По датасету: 
  - можно поискать отделы, реквизиты, новости;
  - персоналии могут быть ликвидированы;
  - ориентир на контенте, размещённом на странице.
