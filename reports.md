# mse1h2026-search

## О проекте

### Постановка задачи

#### Основная проблема

Текущий поиск на сайте etu.ru работает неудовлетворительно — результаты нерелевантны. Использование внешних сервисов (например, API Яндекса) обойдётся примерно в **250 000 рублей в год**, что экономически нецелесообразно.

#### Цель проекта

Провести исследование self-hosted поисковых платформ и разработать backend-сервис (адаптер), который обеспечит интеграцию выбранного решения с сайтом ЛЭТИ.

### Требования

#### Функциональные требования
1.  **Поиск:** Сервис должен предоставлять API для трансляции запросов в поисковую платформу с поддержкой пагинации.
2.  **Фильтрация:** Обеспечение фильтрации результатов по языку (RU/EN) и разделу сайта (области поиска).
3.  **Подсказки:** Реализация механизма автодополнения (autocomplete) на основе индекса или истории запросов.
4.  **Сбор статистики:** Сохранение логов поисковых запросов и фактов перехода по ссылкам (клики).
5.  **Индексация:** Реализация механизма парсинга дампа БД CMS (ModX), очистки данных (фильтрация непубличных/удаленных страниц) и загрузки их в поисковую платформу.

#### Нефункциональные требования
1.  **Архитектура:** Решение должно представлять собой промежуточный сервис (адаптер) над готовой поисковой платформой.
2.  **Развертывание:** Контейнеризация всех компонентов (Docker).
3.  **Стек технологий:**
    *   Backend-сервис: Python (FastAPI/Django).
    *   Поисковая платформа: Self-hosted решение (выбранное на этапе исследования, например Meilisearch, Solr).
    *   Хранилище логов: PostgreSQL.
    *   Фронтенд (демо): Vue.js.
4.  **Ограничения среды:** Оптимизация под сервер с 4 vCPU, 8–16 ГБ RAM, без GPU и с медленным SSD.
5.  **Нагрузка:** Способность обрабатывать запросы от 400–500 активных пользователей в пике.

### Сценарии использования
#### UC-1: Поиск информации пользователем

- **Действующее лицо**: пользователь (анонимный)
- **Предусловия**: 
  - Пользователь открыл демо-интерфейс поиска или сайт etu.ru (в перспективе) и ввёл запрос в строку поиска.
  - В системе проиндексированы данные из дампа: страницы (`document`) и ссылки (`reference`), при этом отфильтрованы невалидные записи (с `published = 0`, `deleted = 1`, `searchable = 0` и т.д.).
- **Основной поток**:
  1. Пользователь вводит запрос (например, «реквизиты», «гранты первокурсникам», «admission requirements») и нажимает Enter.
  2. Система отправляет запрос к Search API (`/api/search?q=<запрос>&lang=ru&start=0&count=20`), где `lang` может определяться автоматически или явно выбираться пользователем.
  3. Поисковый движок возвращает список релевантных документов, учитывая:
     - Мультиязычность (если указан язык, ищутся страницы соответствующего языкового раздела);
     - Для страниц-ссылок (`reference`) возвращается их `final_url` (внешний редирект), для обычных страниц — внутренний URL, построенный на основе алиасов.
  4. Система отображает результаты в демо-интерфейсе (заголовок, краткое описание).
  5. Пользователь кликает на одну из ссылок.
  6. Система логирует клик (запрос, выбранный документ, время, язык) в PostgreSQL.
- **Альтернативы**:
  - Если ничего не найдено, показывается сообщение «Ничего не найдено».
  - Если страница является редиректом, пользователь сразу перенаправляется на внешний ресурс.
- **Постусловия**: пользователь переходит на целевую страницу (внутреннюю или внешнюю).

---

#### UC-2: Поиск с подсказками (autocomplete)

- **Действующее лицо**: пользователь
- **Предусловия**: пользователь начинает вводить запрос в строку поиска
- **Основной поток**:
  1. После ввода 2-3 символов фронтенд отправляет запрос к `/api/suggest?q=<часть_запроса>&lang=...`.
  2. Система возвращает список популярных или релевантных завершений (на основе индекса и истории запросов).
  3. Пользователь выбирает подсказку или продолжает ввод.
- **Постусловия**: выбранная подсказка используется как полный запрос для поиска.

---

#### UC-3: Фильтрация результатов

- **Действующее лицо**: пользователь
- **Предусловия**: пользователь выполнил поиск и хочет уточнить результаты
- **Основной поток**:
  1. После получения результатов пользователь выбирает фильтр (например, язык, раздел сайта, тип документа).
  2. Фронтенд отправляет новый запрос с добавленными параметрами фильтрации (например, `&lang=en`, `&site=abiturientam`).
  3. Система возвращает отфильтрованные результаты.
- **Альтернативы**: фильтры могут быть применены как до, так и после поиска.

---

#### UC-4: Переиндексация данных (администратор)

- **Действующее лицо**: администратор (через запуск скрипта или обращение к внутреннему API)
- **Предусловия**: получен новый дамп базы данных CMS
- **Основной поток**:
  1. Администратор загружает дамп в указанное место и запускает процесс индексации (вручную или по расписанию).
  2. Индексатор читает данные, фильтрует невалидные записи, строит полные URL.
  3. Индексатор отправляет документы в поисковый движок (полная перезапись или обновление).
- **Альтернативы**: возможна инкрементальная индексация, если дамп содержит информацию об изменениях.
- **Постусловия**: поисковая система использует актуальные данные.


### Инструкция по запуску

Клонируем репозиторий:
```bash
git clone https://github.com/moevm/mse1h2026-search.git
```

Переходим в папку с поисковым сервисом:
```bash
cd mse1h2026-search/search-service/
```

Копируем конфиг из примера для backend:
```bash
cp backend/.env.example backend/.env
```

Копируем конфиг из примера для frontend:
```bash
cp frontend/.env.example frontend/.env
```

Запускаем сервис через docker compose:
```bash
docker compose up --build -d
```

После запуска сервисы будут доступны по адресам(при конфиге из `.env.example`):

- Frontend: `http://localhost`
- Backend API: `http://localhost:8000`
- Swagger UI: `http://localhost:8000/docs`
